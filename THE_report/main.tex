\section{Introduction}
\label{Introduction}
%The introduction provides the reader with the reasons for this report and put the report into context. Explain for the reader, the questions that this report tries to cover.



The definition of ”good code” is often debated. Every line of code represents a cost of something, whether it's money or time for proper maintenance and readability. According to Loup Vaillant \cite{vaillant_2015_what} good code is characterized by:
\begin{itemize}
    \item Possibility of writing code as little as possible (less lines of code means less reading and less time spent maintaining)
    \item Code doesn't have many bugs
    \item Code shouldn't use many resources
    \item Code should not perform slow
\end{itemize}

To write a good code programmer should have spent as minimum time as possible on prerequisite knowledge (with what program will do, libraries, frameworks, etc.), knowledge of existing code and what it does, and coordination (working efficiently in team).

This report reflects on a group software development project designed to explore these concepts practically. The project aimed to create a dice game using Object-Oriented Programming (OOP) in Python, using Test Driven Development, unit testing and conventional styles.

This paper covers my experience of doing group project. 
The objectives of this paper are:
\begin{itemize}
    \item Test Driven Development in theory and practise.
    \item Impact of Functional Paradigm on Unit Testing.
    \item Assess code sustainability using cohesion, cyclomatic complexity and maintability index.
\end{itemize}

Where:
\begin{itemize}
    \item h1 stands for \textbf{Unique Operators} ($\eta_1$)
    \item h2 stands for \textbf{Unique Operands} ($\eta_2$)
    \item N1 stands for \textbf{Total Operators} ($N_1$, the total count of all operators used)
    \item N2 stands for \textbf{Total Operands} ($N_2$, the total count of all operands used)
    \item Voc. stands for \textbf{Vocabulary} ($\eta = \eta_1 + \eta_2$, the total number of unique operators and operands)
    \item Len. stands for \textbf{Length} ($N = N_1 + N_2$, the total number of operator and operand occurrences)
    \item Calc. Len. stands for \textbf{Calculated Length} ($\hat{N} = \eta_1 \log_2 \eta_1 + \eta_2 \log_2 \eta_2$, an estimated program length)
    \item Volume stands for \textbf{Volume} ($V = N \log_2 \eta$, the minimum number of bits required to encode the program)
    \item Diff. stands for \textbf{Difficulty} ($D = \frac{\eta_1}{2} \cdot \frac{N_2}{\eta_2}$, a measure of how hard it is to write or understand the program)
    \item Effort stands for \textbf{Effort} ($E = V \times D$, the required mental effort to develop the software).
\end{itemize}

Halstead metrics showed that computer.py required the highest mental effort to develop and understand, with an Effort score significantly higher than other modules (Volume 801.02, Effort 6511.51).

\section{Discussion}

The results of the practical assignment, particularly the low code coverage and
poor cohesion metrics (Section 3), provide a quantitative basis for reflecting on
the efficacy of the chosen development methodologies and their impact on code
sustainability.

\subsection{Philosophies and Principles}
Several philosophies and principles are relevant to the outcome, such as:

\begin{itemize}
    \item DRY(Don't Repeat Yourself) - reduce repetition of information that is likely to change replacing it with abstractions %https://en.wikipedia.org/wiki/Don%27t_repeat_yourself
    \item KISS (Keep it simple stupid) - principle from U.S Navy that all choices should be extremelly simple.
    \item YAGNI (You Ain't Gonna Need It) - way of programming, where things are implemented only when they are required % https://en.wikipedia.org/wiki/You_aren%27t_gonna_need_it
          % https://en.wikipedia.org/wiki/KISS_principle
\end{itemize}
Reflecting personal experience, my approach was aligned closely to KISS: it's preferable to spend extra time to make code shorter and simplify logic. However based on results from Section 3, project work abandoned such principles that led to unnessecary complexity.


\subsection{Test Driven Development and cost of the proccess flaw}
At its core Test Driven Development is about writing tests before the code. Such cycle leads to writing as little code as possible to pass the tests which corresponds to philosophies such as YAGNI and KISS.

\subsubsection{Team's approach to the assignment and how TDD wasn't implemented}
Our team's plan on doing assignment can be described in such steps:
\begin{enumerate}
    \item Create UML and architecture
    \item Assign each person a class.
    \item Each person writes their code.
    \item Person exchanges the code with other member and both of them unit test the other person's code.
    \item After receiving tests and code back, member rewrites their code to pass written tests.
    \item Integrate all classes altogether and then pass the assignment.
\end{enumerate}

Seeing the steps, TDD is not implemented only in half: we started with writing the actual program and not the tests. Team misinterpreted the task and TDD overall, so we wouldn't merge any of the branches until all of them are rightfully tested. This failed applying one of the TDD's leverages: design guidance. Such development of the product leaded for a late introducing of problems. As shown in the  \textbf{Table 6}, bugs that were discovered in post-release cost 10--25$\times$ more if they were taggled in early stages of development. The subsequent "sudden" realisation that "integration needs some time and changes in the code" was an alert of neglecting continuous integration.

\subsection{Functional Programming}
During this project it was discovered that procedural paradigm can cause issues during testing phase, therefore using functional paradigm showed more benefits.

\subsubsection{Pure Functions in Unit Testing}
One of the classes, high\_score.py had functional paradigm in it. Seeing the coverage in Section 3 we can see that only 1 line wasn't tested out of all program. It's one of the most tested files throughout the whole project. The coverage data
(92\%) and high Maintainability Index (81.58) reflect the benefits of this approach

As Functional Programming implements Pure functions, it has a great advantage in unit testing: no side effects in all functions which means same input has the same output all the time. It eliminates dependencies on mostly anything and additionally makes tests simple and very isolated which could be one of properties of clean coding.

\subsubsection{Functional paradigm or conventional style}

During writing the high\_score.py file I encountered into conflict between my vision of functions for unit testing and flake8 view. Specifically, I preferred to pass input variables directly into functions rather than relying on stored class variables. While linters promote conventional OOP encapsulation, in the context of unit testing, the functional preference (requiring explicit input for every function) maximizes testability and readability, aligning with the programming philosophy KISS (Keep It Simple, Stupid) by making dependencies explicit.

Deadline and wrong approach to the project led the team to a technical debt -- quick and simple resolving of the problem that worsened the quality of code and maintability of it. Due to such circumstances testing coverage got extremelly low as there wasn't that much time to retest everything.

\section{Summary}
This paper analyzed the sustainability of a software project through the lens of TDD and functional programming. The key findings are:
\begin{enumerate}
    \item Methodology Matters: Partial adherence to TDD is insufficient. Writing unit tests before integration leads to drastical increases of problems in the code.
    \item Quantitative Metrics reflect Quality: The low cohesion (22.92\%) and coverage (9\%) of the most complex module accurately predicted the difficulty the team faced in writing tests to it.
    \item Functional Paradigm promotes Sustainability: The use of pure functions in \texttt{high\_score.py} resulted in higher maintainability and testability compared to the state-heavy OOP modules.
\end{enumerate}
Sustainable software development requires not just working code, but code that is modular, testable, and developed with a process that catches defects early.

\section{Reflections on Polyglot Programming}
\label{appendix:languages}
During the research phase that were on Canvas, I found an article \cite{a2025_teach} that was debating about books such as "Teach C++ in 24 hours" and how they mislead people into thinking that they will actually know C++ by the end of the book. The core argument is that coding quality and understanding is coming from years of practise, not days. This resonates with the quote: "A language that doesn't affect the way you think about programming is not worth knowing."

Reflecting on this, my mother said that in her university all task had requirement to write on different language. Additionaly, on current course when I had to chance to use Java, I decided to switch sides and write tasks in C/C++. That was a game changer and I incredibly enjoyed it, because I found lots of new information about manipulations with memory that i would have never knew if I used only high-level languages like python or java.

For functional paradigm part of this course, I really badly wanted to write on different language that is based  on this paradigm. Unfortunately, my team didn't allow me to do that. Nevertheless, after spending few hours figuring out what is actually a functional paradigm in OOP Java, I got it. Maybe, it was easier to understand with Java. And functional paradigm changed my thoughts about programming overall. For second assignment I tried to code with same style, which was quite fun. And just because of that I discovered that functional paradigm is perfectly fitted with unit testing. Testing functions is way easier than testing code that has dependances on other functions.
